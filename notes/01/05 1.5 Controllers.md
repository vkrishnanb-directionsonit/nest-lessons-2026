## 1.5 Controllers

In NestJS, **Controllers** are responsible for handling incoming **requests** and returning **responses** to the client. They serve as the entry point to your application's logic.

### 1. The `@Controller()` Decorator
A class becomes a controller when it is annotated with the `@Controller()` decorator. This decorator can take an optional **prefix**, which acts as a base path for all routes defined within that class.

```typescript
@Controller('products') // All routes in this class start with /products
export class ProductsController {
  // ...
}
```

### 2. Routing via HTTP Method Decorators
Inside the controller, you define methods (actions) and associate them with specific HTTP verbs using decorators like `@Get()`, `@Post()`, `@Put()`, `@Delete()`, and `@Patch()`.

```typescript
@Controller('products')
export class ProductsController {

  @Get() // Handles GET /products
  findAll(): string {
    return 'This action returns all products';
  }

  @Get('featured') // Handles GET /products/featured
  findFeatured(): string {
    return 'This action returns featured products';
  }
}
```

### 3. Accessing Request Data
To build dynamic APIs, you need to access data from the incoming request (URL parameters, query strings, or the request body). NestJS provides specific decorators to inject these values directly into your method signatures.

#### **A. Route Parameters (`@Param`)**
Used for identifying a specific resource (e.g., `/products/5`).
```typescript
@Get(':id') // The colon indicates a dynamic parameter
findOne(@Param('id') id: string) {
  return `This action returns product #${id}`;
}
```

#### **B. Query Parameters (`@Query`)**
Used for filtering or pagination (e.g., `/products?sort=desc`).
```typescript
@Get()
findAll(@Query('sort') sortOrder: string) {
  return `Returning products sorted by ${sortOrder}`;
}
```

#### **C. Request Body (`@Body`)**
Used for sending data to the server, typically in POST or PUT requests. This is where you would use a **DTO (Data Transfer Object)** to define the shape of the data.
```typescript
@Post()
create(@Body() createProductDto: any) {
  return 'This action adds a new product';
}
```

### 4. Response Handling and Serialization
One of the most powerful features of NestJS Controllers is how they handle the response:

1.  **Automatic JSON Serialization:** If you return a JavaScript object or array from a controller method, NestJS automatically serializes it to JSON and sets the `Content-Type` header to `application/json`. If you return a string, it sends it as plain text.
2.  **Status Codes:** By default, the response status code is always **200 OK**, except for POST requests, which are **201 Created**. You can override this using the `@HttpCode(204)` decorator.
3.  **Library-specific mode:** If you need manual control over the response (like in Express), you can inject the response object using `@Res()`, but this is generally discouraged as it makes your code platform-dependent and harder to test.

### 5. Dependency Injection in Controllers
Following the SOLID principles, controllers should never contain "business logic" (e.g., database queries or complex calculations). Instead, they **delegate** that work to a Service.

The Service is injected via the **Constructor**:

```typescript
@Controller('products')
export class ProductsController {
  // NestJS looks for AppService in the IoC container and provides the instance here
  constructor(private readonly productsService: ProductsService) {}

  @Get()
  findAll() {
    return this.productsService.getAllProducts();
  }
}
```

### Summary of Controller Responsibility:
*   **Extracts** data from the request (Params, Body, Query).
*   **Validates** that the data is in the correct format (often using Pipes, which we will cover later).
*   **Calls** the appropriate method in a Service.
*   **Returns** the result to the client with the correct HTTP status code.

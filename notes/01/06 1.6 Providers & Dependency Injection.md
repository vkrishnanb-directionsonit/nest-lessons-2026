## 1.6 Providers & Dependency Injection

In NestJS, **Providers** are the fundamental building blocks of the application. While Controllers handle the "outside world" (HTTP), Providers handle the "inside world" (Business Logic). 

The power of Providers lies in **Dependency Injection (DI)**, managed by the NestJS **Inversion of Control (IoC)** container.

---

### 1. What is a Provider?
A provider is simply a class, value, or factory that can be **injected** into other classes. Most of the logic you write—Services, Repositories, Factories, and Helpers—will be categorized as Providers.

The primary identifier of a provider is the **`@Injectable()`** decorator. This decorator provides metadata that allows the Nest IoC container to manage the class.

```typescript
import { Injectable } from '@nestjs/common';

@Injectable()
export class UsersService {
  private readonly users = [{ id: 1, name: 'John Doe' }];

  findAll() {
    return this.users;
  }
}
```

---

### 2. Constructor Injection
NestJS uses **Constructor-based Dependency Injection**. Because you are using TypeScript, Nest resolves dependencies by looking at the types of your constructor arguments.

```typescript
@Controller('users')
export class UsersController {
  // Nest sees 'UsersService' type and looks for it in the IoC container
  constructor(private readonly usersService: UsersService) {}

  @Get()
  findAll() {
    return this.usersService.findAll();
  }
}
```
*Note: The `private readonly` syntax is a TypeScript shorthand that simultaneously declares the property and assigns the injected value to it.*

---

### 3. Custom Providers (The Power of Tokens)
Sometimes, a simple class injection isn't enough. You might want to inject a constant value, a configuration object, or a class that changes based on the environment. NestJS supports **Custom Providers**.

Instead of just listing a class in the `providers` array, you can use an object with a **provide** token and a **use** strategy:

#### **A. Value Providers (`useValue`)**
Used for constant values, configuration objects, or external libraries.
```typescript
@Module({
  providers: [
    {
      provide: 'CONNECTION_STRING',
      useValue: 'mongodb://localhost:27017',
    },
  ],
})
```
To inject a non-class token, you must use the **`@Inject()`** decorator:
```typescript
constructor(@Inject('CONNECTION_STRING') private connection: string) {}
```

#### **B. Factory Providers (`useFactory`)**
Used when the provider is dynamic or needs complex initialization (e.g., a database connection that requires other services).
```typescript
{
  provide: 'ASYNC_CONNECTION',
  useFactory: async (configService: ConfigService) => {
    const connection = await createConnection(configService.getDbConfig());
    return connection;
  },
  inject: [ConfigService], // Dependencies for the factory
}
```

#### **C. Class Providers (`useClass`)**
Used for **Polymorphism**. You can define an abstract token and swap the implementation (e.g., using a `MockLogger` for tests and a `RealLogger` for production).

---

### 4. Optional Providers
Sometimes a dependency is "nice to have" but not strictly required for the class to function. You can mark a dependency as optional to prevent Nest from throwing an error if it isn't found in the container.

```typescript
import { Injectable, Optional, Inject } from '@nestjs/common';

@Injectable()
export class MyService {
  constructor(@Optional() @Inject('HTTP_OPTIONS') private httpClient: any) {}
}
```

---

### 5. Global Providers
Normally, providers are scoped to the module they are defined in. If you want a provider to be available everywhere in the application without having to import its module into every other module, you can use the **`@Global()`** decorator on the module.

*Caution: Global modules should be used sparingly (e.g., for Config, Database, or Logging) to avoid cluttering the global namespace.*

---

### 6. The Provider Lifecycle & Management
When the NestJS application starts:
1.  **Scanning**: Nest reads all `@Module()` decorators.
2.  **Dependency Resolution**: Nest creates a "topological sort" of your providers. If `Service A` needs `Service B`, Nest ensures `Service B` is created first.
3.  **Instantiation**: Providers are instantiated (by default as Singletons).
4.  **Injection**: The instances are passed into the constructors of the classes that require them.

### Summary
*   **Providers** = Classes/Values that hold logic or data.
*   **`@Injectable()`** = Marks a class as manageable by Nest.
*   **Tokens** = The "Key" used to look up a provider (can be a Class name or a String/Symbol).
*   **IoC Container** = The "Brain" that holds the instances and handles the "new" keyword for you.

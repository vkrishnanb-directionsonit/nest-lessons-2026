# NestJS - main three main "pillars."

3 March 2026
Current Version: 11

NestJS was designed specifically to bring the enterprise-grade patterns **(SOLID, DI, Decorators)** to the Node.js ecosystem.

In NestJS, we organize everything into three main "pillars."

### 1. Modules (`@Module`)
Think of these as the **organizational boundaries**. Every NestJS app has at least one (the "Root Module").
You might think of them as a way to group a "Feature" or a logical "Namespace" that explicitly declares its dependencies.
*   **Key Role:** They encapsulate a set of related Controllers and Providers. If a Controller wants to use a Service, they usually must belong to the same Module (or be explicitly exported/imported).

### 2. Controllers (`@Controller`)
These are your **API entry points**.
*   **Key Role:** They handle incoming **Requests** and return **Responses**. They should be "lean"—delegating heavy lifting to services.  
This class is the **"Network Boundary."** Its job is to listen to the internet, validate the incoming request, and return a response. If you don't want a route to be public, you don't put it in a Controller.  
In NestJS, there is no such thing as a **"Private Controller."** If a class is intended to be "internal only" and not exposed to the outside world via a URL or endpoint, it should not be a Controller at all. Instead, it should be a Provider (typically a Service).
### 3. Providers (`@Injectable`)
This is the most broad category. Almost everything else is a "Provider": Services, Repositories, Factories, etc.
*   **Key Role:** They contain the **Business Logic**.
*   **The Magic:** They are managed by the NestJS **IoC (Inversion of Control) container**.  
This class (usually a Service) contains your business logic. This is where you put code that is "accessed only within the app."
---
### 4. Encapsulation in the `@Module`
NestJS provides a powerful encapsulation system, but it applies to **Providers**, not Controllers.

Here is how the visibility works within a Module:

| Property | Visibility / Access |
| :--- | :--- |
| **`controllers`** | **Public to the Web.** Any class here is automatically "exposed" as a set of API endpoints. |
| **`providers`** | **Private by default.** Classes here (like `AppService`) can only be seen and used by other classes *within the same module*. They are hidden from the rest of the application. |
| **`exports`** | **The Public API of the Module.** If you want a Service from Module A to be usable inside Module B, you must explicitly list that Service in the `exports` array of Module A. |

---
**If you have two separate modules—`UsersModule` and `OrdersModule`—and the `OrdersController` needs to call a method in the `UsersService`, how do you think NestJS requires you to "share" that service between them?** (Think about "encapsulation" vs "public API").

### The "Under the Hood" Libraries
By default, NestJS doesn't reinvent the wheel for the HTTP layer. It acts as an abstraction over:
1.  **Express 5** (The default in v11): Robust and widely used.
2.  **Fastify 5** (The high-performance alternative): You can swap to this if you need extreme throughput.  

It also relies heavily on **`reflect-metadata`** to make those decorators work, similar to how C# uses Attributes and Reflection.

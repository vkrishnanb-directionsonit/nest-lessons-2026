# 2.1 Pipes 

Welcome to **Phase 2.1: Pipes**.

### **Overview: The Request Pipeline Position**
In the NestJS Request Lifecycle, Pipes execute **after** Guards and Interceptors, but **before** the Controller's route handler.

> **Execution Order:**  
> `Request` ➡️ `Middleware` ➡️ `Guards` ➡️ `Interceptors (Pre)` ➡️ **`Pipes`** ➡️ `Controller Handler`

---
In NestJS, **Pipes** are specialized classes used to process data before it reaches your route handler. Think of them as a "filter" or "translator" that sits between the incoming request and your application logic.

### **1. The Two Main Roles of Pipes**
Pipes implement the `PipeTransform<T, R>` interface and serve two distinct purposes:

1.  **Transformation:** Changing the input data to a different format (e.g., converting a string `"5"` from a URL into an actual number `5`).
2.  **Validation:** Checking if the input data is correct. If the data doesn't meet your rules, the pipe stops the request and sends back an error message before your code even runs.

#### **A. Transformation (Data Marshalling)**
Converts input data to the desired type. For example, converting a URL string parameter `:id` into a JavaScript `number`.
**Built-in Transformation Pipes**
NestJS provides several ready-to-use pipes for common data conversions. These are most often used with URL parameters (`@Param`) or query strings (`@Query`).

*   `ParseIntPipe`: Converts a string to an integer.
*   `ParseBoolPipe`: Converts a string to a boolean.
*   `ParseUUIDPipe`: Validates that a string is a valid UUID.
*   `ParseArrayPipe`: Parses an incoming string into an array.

**Example Usage:**
```typescript
@Get(':id')
findOne(@Param('id', ParseIntPipe) id: number) {
  // If 'id' is "abc", NestJS automatically returns a 400 Bad Request.
  // If 'id' is "123", it arrives here as the number 123.
  return this.service.getById(id);
}
```
---
#### **B. Validation (Schema Enforcement)**

Evaluates input data; if valid, it passes through unchanged. If invalid, it throws an exception (usually a `400 BadRequestException`) before your controller logic even runs.
*   **Built-in example:** `ValidationPipe`.  

**Validation with DTOs**  
In professional applications, we use **DTOs (Data Transfer Objects)** to define the shape of the data we expect in a request (like a POST body). 
To make pipes work for validation, we use two standard libraries: `class-validator` and `class-transformer`.

#### **Step A: Define the DTO**
You use decorators to set the rules for each property.

```typescript
import { IsString, IsInt, MinLength, Max, IsEmail } from 'class-validator';

export class CreateUserDto {
  @IsString()
  @MinLength(3)
  username: string;

  @IsEmail()
  email: string;

  @IsInt()
  @Max(120)
  age: number;
}
```

#### **Step B: Apply the ValidationPipe**
In your controller, you tell NestJS to use the `ValidationPipe` for the incoming body.

```typescript
@Post()
create(@Body(new ValidationPipe()) createUserDto: CreateUserDto) {
  return this.userService.save(createUserDto);
}
```

---
### **3. Setting up Global Validation**
Instead of adding `new ValidationPipe()` to every single route, you can set it up once for your entire application in the `main.ts` file. This is the recommended "enterprise" approach.

```typescript
// main.ts
import { ValidationPipe } from '@nestjs/common';

async function bootstrap() {
  const app = await NestFactory.create(AppModule);
  
  app.useGlobalPipes(new ValidationPipe({
    whitelist: true,               // Automatically remove any properties not defined in the DTO
    forbidNonWhitelisted: true,    // Throw an error if the user sends properties not in the DTO
    transform: true,               // Automatically convert incoming types to match the DTO types
    transformOptions: {
    enableImplicitConversion: true, // Automagically convert strings to numbers/booleans in DTOs
    },
  }));

  await app.listen(3000);
}
```

### **4. Creating a Custom Pipe**
If the built-in pipes aren't enough, you can create your own. Every pipe must implement the `PipeTransform` interface.

```typescript
import { PipeTransform, Injectable, ArgumentMetadata, BadRequestException } from '@nestjs/common';

@Injectable()
export class CustomValidationPipe implements PipeTransform {
  transform(value: any, metadata: ArgumentMetadata) {
    // 'value' is the incoming data
    // 'metadata' contains info about the data (is it a Body, Param, or Query?)
    
    if (value === 'secret-admin-code') {
      return value;
    }
    
    throw new BadRequestException('The provided code is invalid');
  }
}
```

```typescript
import { PipeTransform, Injectable, ArgumentMetadata, BadRequestException } from '@nestjs/common';

@Injectable()
export class ParseEntityPipe implements PipeTransform {
  // value: The raw input
  // metadata: Info about the target (body, query, param, etc.)
  transform(value: any, metadata: ArgumentMetadata) {
    const isNumeric = !isNaN(parseFloat(value)) && isFinite(value);
    if (!isNumeric) {
      throw new BadRequestException('Validation failed: ID must be a number');
    }
    return parseInt(value, 10);
  }
}
```

---

### **Summary of Benefits**
*   **Cleaner Controllers:** Your logic doesn't get cluttered with `if (typeof id !== 'number')` checks.
*   **Consistency:** Errors are returned in a standard format throughout your app.
*   **Security:** Using the `whitelist` feature prevents "Mass Assignment" attacks where users try to send extra data (like `isAdmin: true`) that your DTO doesn't allow.
